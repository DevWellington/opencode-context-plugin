# Plan 05-06: Auto-Generate Reports on Session End

## Objective

After every `/exit` or `/compact`, automatically regenerate:
- Monthly report (`reports/monthly-YYYY-MM.md`)
- Annual report (`reports/annual-YYYY.md`)
- Full `intelligence-learning.md` with extracted patterns/bugs

## Problem Statement

Currently `saveContext()` only:
- Updates metadata in `intelligence-learning.md` (timestamp, session count)
- Updates daily/week/day summaries
- Does NOT regenerate monthly/annual reports
- Does NOT do full intelligence extraction with patterns/bugs

## Solution

Add non-blocking calls to existing agent functions after context save completes. Use aliased imports to avoid naming conflict between `./intelligence.js` (metadata-only) and `../agents/generateIntelligenceLearning.js` (full extraction).

## must_haves

**Truths:**
- After every `/exit` or `/compact`, monthly report is regenerated
- After every `/exit` or `/compact`, annual report is regenerated
- After every `/exit` or `/compact`, full intelligence-learning.md is regenerated with patterns/bugs
- Report generation failures do NOT block or slow down context save

**Artifacts:**
- `src/modules/saveContext.js` - Modified to call report generation functions
- `src/agents/generateMonthly.js` - Already exists, used via import
- `src/agents/generateAnnual.js` - Already exists, used via import
- `src/agents/generateIntelligenceLearning.js` - Already exists, used via import

**Key Links:**
- `saveContext()` → `generateMonthlySummary()` (non-blocking)
- `saveContext()` → `generateAnnualSummary()` (non-blocking)
- `saveContext()` → `updateIntelligenceLearning()` from agents (full extraction)

### Task 2: Add Non-Blocking Report Regeneration Calls

**Files:** `src/modules/saveContext.js`

**Action:**
After the search index update (around line 140), add a new section that triggers report regeneration. This must be non-blocking (fire-and-forget with error handling) to not slow down session save:

```javascript
// Regenerate reports (non-blocking, best-effort)
// These run after context save completes without blocking
setImmediate(async () => {
  try {
    const now = new Date();
    const year = now.getFullYear();
    const month = `${year}-${String(now.getMonth() + 1).padStart(2, '0')}`;

    // Run all regenerations in parallel
    await Promise.all([
      generateMonthlySummary(directory, month).catch(err => {
        logger(`[saveContext] Monthly report generation failed (non-fatal): ${err.message}`);
      }),
      generateAnnualSummary(directory, year).catch(err => {
        logger(`[saveContext] Annual report generation failed (non-fatal): ${err.message}`);
      }),
      updateIntelligenceLearning(directory).catch(err => {
        logger(`[saveContext] Intelligence learning regeneration failed (non-fatal): ${err.message}`);
      })
    ]);

    logger(`[saveContext] Report regeneration complete`);
  } catch (error) {
    // Outer catch prevents unhandled rejection but doesn't block
    logger(`[saveContext] Report regeneration error: ${error.message}`);
  }
});
```

**Note:** The existing `await updateIntelligenceLearning(directory, learningSessionInfo)` at line 131 should remain for metadata-only update. The new `updateIntelligenceLearning(directory)` from agents does full extraction including patterns/bugs.

**Verify:**
```bash
node -e "import('./src/modules/saveContext.js').then(m => console.log('saveContext exports:', typeof m.saveContext))"
```

**Done:** Report regeneration calls added, non-blocking via setImmediate

## Why setImmediate Instead of Await?

Using `setImmediate()` instead of `await` ensures:
1. Context save returns immediately to caller
2. Report generation happens after current event loop tick
3. Failures don't affect the context save outcome
4. Similar pattern to how search index update is already handled (try/catch, non-fatal)

## Alternative Considered: Debouncing

We could add debouncing to avoid excessive regenerations if `/compact` is called rapidly. However:
- The agent functions themselves handle their own internal state
- `generateMonthlySummary` and `generateAnnualSummary` scan sessions and regenerate fully each time
- If user calls `/compact` 10 times rapidly, we regenerate 10 times (acceptable trade-off)

If debouncing is later needed, it can be added via a simple debounce utility, but premature optimization is not warranted.
